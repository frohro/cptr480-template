### Overview

The PCM1808 is a 24-bit stereo audio ADC connected to the YD-RP2040 via I2S (the 3-wire serial audio interface: SCK, LRCK, DATA). The RP2040 has no dedicated I2S peripheral, so you will implement I2S receive using a PIO state machine — exactly the approach the Pico SDK's audio libraries use internally.

After this lab, raw 32-bit words (containing 24-bit audio samples from the PCM1808) will flow from PIO → DMA → memory continuously, without any CPU intervention between samples. This DMA pipeline is the foundation that all subsequent audio labs build on.

## Part 1 — PCM1808 Hardware Check (~20 min)

Before writing firmware, verify the PCM1808 is properly powered and its mode pins are configured correctly for Standard I2S mode.

1. From the 2026 board schematic, determine:
   - The PCM1808 power pin voltages (VDD and VDDA — are they both powered?)
   - The logic levels on FMT, MD0, MD1 (tied high/low to 3.3 V? connected to a GPIO?)
   - Which Pico GPIOs connect to BCK, LRCK (WS), and DOUT
   - The SCKI source: is it from the Si5351a, a dedicated CMOS oscillator, or generated by PIO?

2. Measure VDD and VDDA at the PCM1808 with a multimeter. Record in `lab4/lab4_report.md`.

3. Probe the SCKI line with the logic analyzer. Is it oscillating? At what frequency?
   Determine the PCM1808 operating mode from MD0 and MD1 (datasheet Table 2). The board
   is wired for **Master 512fs** mode, so SCKI = 512 × fs. At fs = 48 kHz this gives
   SCKI = 24.576 MHz. In master mode the PCM1808 *outputs* BCK (3.072 MHz) and LRCK
   (48 kHz) — the RP2040 PIO is the slave receiver.

4. Identify which GPIO is connected to FMT and what the firmware must drive it to in order
   to select I2S standard 24-bit format. (Hint: FMT = LOW.)

> If SCKI is not present or is the wrong frequency, fix this first — the PCM1808 will not
> generate BCK or LRCK without a correct SCKI, and the PIO will hang waiting for clock edges.

## Part 2 — PIO I2S Receive Program (~45 min)

Create `lab4/i2s_rx.pio` — the PIO state machine program for I2S receive.

The I2S protocol (from the PCM1808's perspective):
- **LRCK** (word clock): LOW = left channel, HIGH = right channel
- **BCK** (bit clock): data is presented on BCK falling edge, stable on rising edge
- **DATA**: MSB first, 24 valid bits, zero-padded to 32 bits per frame

Your PIO program must:
1. Wait for LRCK to go LOW (start of left channel)
2. Clock in 32 bits from DATA on BCK rising edges (using `in pins, 1` and `push`)
3. Repeat for the right channel HIGH phase

The `pico-extras` repository contains a reference `audio_i2s.pio` — you may study it but write your own (or understand it well enough to adapt it with Copilot). Note that the existing programs typically implement I2S *transmit* (master); you need *receive* (slave, where the PCM1808 provides BCK and LRCK).

> **Key detail:** The PCM1808 is I2S master by default — it drives BCK and LRCK. The RP2040 PIO is the slave, receiving on those clocks. Your PIO program must be clock-synchronous to the PCM1808's BCK.

Commit your PIO program and a brief comment block explaining each instruction.

## Part 3 — DMA Ping-Pong Setup (~40 min)

Create `lab4/main.c` implementing DMA ping-pong for continuous capture:

```
Buffer A [96 × 2 × 4 bytes = 768 bytes]  ← DMA channel 0 fills this
Buffer B [96 × 2 × 4 bytes = 768 bytes]  ← DMA channel 1 fills this
```

(96 samples per channel per half-buffer = 2ms at 48 kHz — a reasonable starting size)

DMA channel 0 and channel 1 are chained: when channel 0 completes (Buffer A full), it triggers channel 1 to start (filling Buffer B), and raises an interrupt. The interrupt handler processes Buffer A while DMA fills Buffer B. When channel 1 completes, it chains back to channel 0.

Key Pico SDK functions:
- `dma_channel_get_default_config()`
- `channel_config_set_transfer_data_size()` — use `DMA_SIZE_32`
- `channel_config_set_read_increment()` / `channel_config_set_write_increment()`
- `channel_config_set_dreq()` — set to the PIO RX DREQ so DMA waits for PIO data
- `channel_config_set_chain_to()` — chain A→B and B→A
- `dma_channel_set_irq0_enabled()` — enable interrupt on completion

**Important:** Configure and arm the DMA channels in `main()` but do **not** start them
yet and do **not** enable the IRQ yet. Both of those happen in `core1_entry()` (Part 5)
so that `DMA_IRQ_0` is registered on Core 1's NVIC, not Core 0's. This keeps Core 0's
interrupt table free for TinyUSB in Lab 7.

Also: the ISR must reset each completed channel's write-address and transfer-count after
it fires — the chain mechanism will re-trigger the channel using whatever register values are
present, so those must be restored to the buffer start before the partner channel finishes.
Do **not** use `memcpy` inside the ISR; just fix the registers and pass the pointer on.

> **Deliverable:** Logic analyzer showing the DMA interrupt GPIO toggling at ~500 Hz (once per 2ms half-buffer completion). Save as `lab4/dma_timing.png`.

## Part 4 — Core Assignment: Core 1 Owns the DMA ISR (~25 min)

The RP2040 has two independent NVIC interrupt controllers — one per core. An IRQ fires on
whichever core registered it with `irq_set_exclusive_handler()`. **This matters:** in Lab 7,
Core 0 will run `tud_task()` in a tight loop. If `DMA_IRQ_0` were registered on Core 0,
TinyUSB's interrupt latency and the DMA ISR would interfere with each other. By registering
the DMA IRQ on Core 1 now, we avoid this problem in every future lab.

Implementation:

1. In `main()` (Core 0): configure both DMA channels fully (Part 3) but do **not** call
   `dma_channel_set_irq0_enabled()`, `irq_set_exclusive_handler()`, or `dma_channel_start()`.
   Then call `multicore_launch_core1(core1_entry)`.

2. In `core1_entry()` (running on Core 1):
   - Drain the inter-core FIFO (`multicore_fifo_drain()`) to clear any boot residue.
   - Enable IRQ signalling on both channels (`dma_channel_set_irq0_enabled()`).
   - Register `DMA_IRQ_0` handler on Core 1 (`irq_set_exclusive_handler()` +
     `irq_set_enabled()`).
   - Start DMA channel A (`dma_channel_start()`).
   - Loop forever in `tight_loop_contents()` — all work is interrupt-driven.

3. The DMA ISR (on Core 1) posts the filled buffer pointer to Core 0 using
   `multicore_fifo_push_timeout_us(ptr, 0)` — **non-blocking** (timeout = 0 µs).
   If Core 0 is slow and the FIFO is full, the buffer is silently dropped rather than
   stalling the ISR and causing a DMA overrun.

4. Core 0's `main()` while-loop calls `multicore_fifo_pop_timeout_us(&ptr, timeout)` and
   prints the samples (Part 5). Core 0 is now structurally equivalent to the Lab 7 USB task
   loop — replacing the print with `tud_task()` is the only change needed in Lab 7.

> **Why this matters:** In the final SDR, Core 0 runs TinyUSB with hard real-time isochronous
> transfer deadlines. Core 1 owns the DMA ISR and delivers audio buffer pointers to Core 0
> via the hardware FIFO. The `tud_audio_tx_done_pre_load_cb()` callback pops the pointer and
> calls `tud_audio_write()` — the DMA/Core 1 side is **unchanged** between Lab 4 and Lab 7.

## Part 5 — Audio Data Verification (~40 min)

Now print actual audio sample values.

The DMA ISR (running on Core 1 after Part 4) posts the filled buffer pointer to Core 0 via
the inter-core hardware FIFO. Core 0 receives it and prints the samples. The architecture
is already in place from Part 4; here you just verify the data is correct:

1. In Core 0's consumer loop, call `multicore_fifo_pop_timeout_us(&ptr, timeout)` to receive
   the buffer pointer. Print the first 8 stereo pairs as sign-extended 24-bit integers:
   ```c
   int32_t left  = (int32_t)buf[2*i]     >> 8;   // sign-extend from bit 31
   int32_t right = (int32_t)buf[2*i + 1] >> 8;
   ```
   The PCM1808 delivers 24 valid bits in bits [31:8]; bits [7:0] are always zero.
   Right-shifting the whole word by 8 as `int32_t` correctly sign-extends the sample.
   **Do not shift before passing to USB** — Linux `snd-usb-audio` expects the sample
   left-justified in the 32-bit word (bits [31:8]), which is exactly what the PIO delivers.

2. Expected output when the audio input is silent: values should be near zero (within a few
   LSBs of the PCM1808's noise floor — expect a few hundred counts at most, not thousands).

3. Expected output with a 1 kHz tone plugged into the audio input: values should oscillate
   sinusoidally between positive and negative peaks.

4. Capture the serial output for both cases and commit as `lab4/silent_output.txt` and
   `lab4/tone_output.txt`.
